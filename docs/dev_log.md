## What This Is

A fully local, zero-cost personal knowledge system. Two tiers:

- **Tier 1 — Active Brain**: Obsidian vault. Human writes notes here. Fast, daily use. Organized with PARA (Projects / Areas / Resources / Archives) + Zettelkasten atomic notes.
- **Tier 2 — Archive Brain**: LightRAG knowledge graph. AI indexes completed work here. Hybrid search: BM25 + vector embeddings + entity graph.

The system feeds itself. Research papers generated by AutoResearchClaw get indexed into LightRAG, which then answers questions about their own findings. The loop is: question → research → paper → archive → graph → better answers → new question.

No cloud APIs. No recurring costs. Everything runs on local hardware via Ollama.

## How AutoResearchClaw Feeds the Brain

AutoResearchClaw is a 23-stage autonomous research pipeline:

1. **Stages 1–9**: Literature search (arXiv, Semantic Scholar), screening, knowledge extraction, synthesis, hypothesis generation.
2. **Stages 10–15**: Experiment design, code generation (via Qwen), resource planning, experiment run, iteration, result analysis.
3. **Stages 16–23**: Paper outline, draft, peer review, revision, quality gate, knowledge archive, export, citation verify.

After a sprint completes, the deliverables (`paper_final.md`, `paper.tex`, `references.bib`, knowledge cards) land in `autoresearchclaw/artifacts/<run-id>/deliverables/`. The paper draft itself is at `artifacts/<run-id>/stage-17/paper_draft.md`.

**As of 2026-04-29**, the manual copy step is replaced by the validation harness:
```powershell
python scripts/validate_and_archive.py --artifact autoresearchclaw/artifacts/rc-<run-id>/
```
This extracts claims, validates them via Ollama (qwen3.5:4b), writes `knowledge_base/system/review-queue.jsonl`, enriches the paper with a `validation_summary:` frontmatter block, copies it to `4. Archives/`, and triggers incremental LightRAG indexing automatically. See `docs/validation-harness.txt` for the full spec.

**Sprint 1 topic**: Survey of cross-encoder reranking for hybrid BM25 + dense retrieval on CPU with sub-2-second latency. This directly feeds back into improving `query.py`.

## Bugs Encountered and Root Causes

This section is the important one. Every bug below burned real time. Don't repeat them.

---

### Bug 1 — `UnicodeEncodeError` on Windows

**Symptom**: Any script with emoji or arrow characters (`→`, `✅`) crashes with `charmap codec can't encode character`.

**Root cause**: Windows terminal defaults to `cp874` or `cp1252` encoding. Python's print tries to encode to that.

**Fix**: Set encoding before running anything:
```powershell
$env:PYTHONIOENCODING="utf-8"
```

**Note**: Linux syntax (`PYTHONIOENCODING=utf-8 python ...`) does not work in PowerShell. Must use `$env:`.

---

### Bug 2 — Gemini CLI Disconnects on Long LLM Calls

**Symptom**: AutoResearchClaw Stage 10 (code generation, ~300s) fails with:
```
ACP prompt failed (exit 1): Queue owner disconnected before prompt completion
```

**Root cause**: Gemini CLI v0.37.1 drops the ACP session connection during calls longer than ~332 seconds. The `acpx` bridge (`acp_client.py`) had no handler for this specific error string.

**Fix 1**: Added `"Queue owner disconnected"` to `_RECONNECT_ERRORS` in `researchclaw/llm/acp_client.py`. Also bumped `_MAX_RECONNECT_ATTEMPTS` from 2 to 3.

**Fix 2 (more reliable)**: Switched the LLM backend to Qwen via Ollama for long calls. Ollama is local and persistent — it never drops connections. Set in `config.arc.yaml`:
```yaml
llm:
  provider: "openai-compatible"
  base_url: "http://localhost:11434/v1"
  primary_model: "qwen3.5:4b"
```

### Bug 3 — Stage 17 Paper Draft Blocked: "No Metrics"

**Symptom**:
```
BLOCKED: Cannot write paper — experiment produced NO metrics
```

**Root cause**: AutoResearchClaw requires experiment metrics before allowing paper writing. Survey/review topics don't produce metrics — they synthesize existing literature. The pipeline had no bypass for this.

**Fix**: AutoResearchClaw has a function `_topic_is_literature_first()` that checks if the topic string contains "Survey" or "review". Prepend your topic with "Survey of" in `config.arc.yaml` and the metrics gate is bypassed.

```yaml
research:
  topic: "Survey of cross-encoder reranking for hybrid BM25..."
```

### Bug 4 — LightRAG Embedding Dimension Mismatch (768 vs 1024)

**Symptom**:
```
ValueError: Embedding dimension mismatch detected: total elements (768)
cannot be evenly divided by expected dimension (1024).
```
Happens on every document. Even after wiping the `.lightrag` directory.

**Root cause**: `lightrag.llm.ollama.ollama_embed` is decorated with `@wrap_embedding_func_with_attrs(embedding_dim=1024, ...)` — hardcoded for `bge-m3:latest`. When called, it validates that the returned embeddings are 1024-dimensional. `nomic-embed-text` returns 768-dim vectors, which fails this check.

This is a **LightRAG library bug** (or opinionated default) — the decorator on `ollama_embed` silently overrides whatever dimension you configure in your `EmbeddingFunc`.

**What does NOT fix it**:
- Wiping the `.lightrag` directory (the error is in the live call, not stored data)
- Setting `embedding_dim=768` in your `EmbeddingFunc` wrapper (the inner decorated function validates first)

**Fix**: Bypass `ollama_embed` entirely. Call the Ollama client directly in `index_archive.py`:

```python
# BEFORE (broken):
from lightrag.llm.ollama import ollama_model_complete, ollama_embed

async def _local_embed(texts):
    return await ollama_embed(texts, embed_model=EMBED_MODEL, host=OLLAMA_HOST)

# AFTER (correct):
from lightrag.llm.ollama import ollama_model_complete  # ollama_embed removed

async def _local_embed(texts):
    import ollama
    import numpy as np
    client = ollama.AsyncClient(host=OLLAMA_HOST)
    data = await client.embed(model=EMBED_MODEL, input=texts)
    return np.array(data["embeddings"])
```

The outer `EmbeddingFunc(embedding_dim=768, func=_local_embed)` then validates correctly.

### Bug 5 — LLM Worker Timeout During Indexing

**Symptom**:
```
WARNING: LLM func: Worker timeout for task ... after 360s
ERROR: Failed to extract entities and relationships
lightrag.utils.WorkerTimeoutError: Worker execution timeout after 360s
```

**Root cause**: LightRAG's extraction prompt has a fixed 1305-token system prompt overhead. At the original `CHUNK_MAX_CHARS=4000`, each chunk produces ~4463 total tokens (input + extracted output). At ~15 tok/s, that's ~370 seconds — just over the 360s worker timeout.

**Calculation**:
```
Total tokens = prompt_overhead (1305) + content_tokens (chunk_chars / 3.8) + output_tokens (content * 3x)
At 15 tok/s: 4000 chars => ~370s  TIMEOUT
At 15 tok/s: 1500 chars => ~186s  OK
```

**Fix**: Reduce chunk size in `config.py`:
```python
CHUNK_MAX_CHARS = int(os.getenv("BRAIN_CHUNK_SIZE", "1500"))
```

**Also**: Qwen 3.5:4b was defaulting to `num_ctx=262144` (its advertised max). Ollama was allocating a massive KV cache, leaving little VRAM for fast inference. Set it to what you actually need:
```python
LOCAL_CONTEXT_WINDOW = int(os.getenv("BRAIN_CONTEXT_WINDOW", "4096"))
```
This is passed through to Ollama via `options: {num_ctx: 4096}` and frees up significant VRAM.

**Enable flash attention** before starting Ollama:
```powershell
$env:OLLAMA_FLASH_ATTENTION="1"
```

### Bug 8 — `ollama_model_complete` Cannot Be Called Directly

**Symptom**:
```
KeyError: 'hashing_kv'
```
When calling `ollama_model_complete` from `lightrag.llm.ollama` outside of LightRAG.

**Root cause**: `ollama_model_complete` is an internal LightRAG function. It expects a `hashing_kv` object injected by the LightRAG framework — it's not a standalone Ollama client wrapper.

**Fix**: For direct Ollama calls (testing, scripts), use `ollama.AsyncClient` directly:
```python
import ollama
client = ollama.AsyncClient(host=OLLAMA_HOST)
response = await client.chat(
    model=LOCAL_LLM_MODEL,
    messages=[{"role": "user", "content": prompt}],
    think=False,
    options={"num_ctx": LOCAL_CONTEXT_WINDOW},
)
result = response["message"]["content"]
```

### Bug 7 — Qwen 3.5 Thinking Mode Silently Kills Indexing Speed

**Symptom**: Even a single small chunk (1/1) times out at 360s. Reducing chunk size to 1500 chars makes no difference. The model just takes forever regardless of content length.

**Root cause**: Qwen 3.5 has a "thinking" capability (visible in `ollama show qwen3.5:4b` under Capabilities). Before generating any response, it produces a long internal chain-of-thought (`<think>...</think>`) that can be 2000+ tokens. This thinking trace is invisible in the output but costs full generation time. For a knowledge extraction task, thinking adds zero quality — it's designed for math and logic problems — but it burns 3-6 minutes per chunk.

**What does NOT fix it**:
- Reducing chunk size (the overhead is in the thinking trace, not the content)
- `PARAMETER think false` in the Ollama Modelfile (not a valid parameter — will error)

**Fix**:

Step 1 — Create a custom Ollama model variant with correct context window:
```powershell
@"
FROM qwen3.5:4b
PARAMETER num_ctx 4096
PARAMETER num_gpu 99
"@ | Out-File -FilePath "$env:TEMP\Modelfile" -Encoding utf8

ollama create qwen3.5:4b-brain -f "$env:TEMP\Modelfile"
```

Step 2 — Pass `think: False` via the API options in `index_archive.py`:
```python
"kwargs": {"host": OLLAMA_HOST, "options": {"num_ctx": LOCAL_CONTEXT_WINDOW, "think": False}},
```

Step 3 — Update `config.py` to use the new model:
```python
LOCAL_LLM_MODEL = os.getenv("BRAIN_LOCAL_MODEL", "qwen3.5:4b-brain")
```

**Why two steps**: The Modelfile bakes in `num_ctx=4096` permanently (Qwen's default is 262144 — Ollama allocates KV cache for the full context window, destroying VRAM headroom). The `think: False` option disables the reasoning trace at the API call level, which is where Ollama exposes that toggle.

**Side effect — event loop error on timeout**: When the LLM worker times out and the process is killed, you may see:
```
ERROR | Task was destroyed but it is pending!
ERROR: LLM func: Critical error in worker: Event loop is closed
RuntimeError: no running event loop
```
This is a cleanup artifact from LightRAG's async worker pool shutting down mid-task. It is cosmetic — it does not indicate data corruption. The real error is the timeout that preceded it.

### Bug V1 — `locate_artifacts` Looks at Artifact Root, Not `stage-17/`

**Symptom**: `FileNotFoundError: Missing artifact file(s): ['...paper_draft.md']` on every real AutoResearchClaw artifact.

**Root cause**: The spec assumed `paper_draft.md` would be at the artifact root. In the actual AutoResearchClaw layout, it lives in `stage-17/`. The artifact root only contains `pipeline_summary.json`, `checkpoint.json`, `heartbeat.json`, and stage directories.

**Fix**: `locate_artifacts` now checks root first, then `stage-17/` as a fallback:
```python
paper_candidates = [
    artifact_dir / "paper_draft.md",
    artifact_dir / "stage-17" / "paper_draft.md",
]
paper = next((p for p in paper_candidates if p.exists()), None)
```

---

### Bug V2 — Section Regex Doesn't Match Numbered Headings

**Symptom**: 0 claims extracted on every real paper. Gemini fallback ran with `paper_text[:800]` (dense method-section math) instead of the abstract/conclusion, returning empty or useless results.

**Root cause**: AutoResearchClaw papers use numbered headings: `## 10. Conclusion`, `## 1. Introduction`. The extraction regex `r'^#{1,3}\s*conclusion'` requires the heading to start with the word directly — it doesn't match when a section number precedes it.

**Fix**: Allow an optional number prefix in all section regexes:
```python
r"^#{1,3}\s*(\d+\.\s*)?conclusion"
r"^#{1,3}\s*(\d+\.\s*)?abstract"
```
Applied in both `_extract_from_llm()` and the context-building block in `run_pipeline()`.

---

### Bug V3 — `gemini-1.5-flash` Returns 404

**Symptom**: `404 Client Error: Not Found` on every Gemini call. Claims extraction silently returns `[]`.

**Root cause**: `gemini-1.5-flash` has been deprecated and returns 404 via the REST API. The default in `config.py` was never updated.

**Fix**: Set in `.env`:
```
BRAIN_GEMINI_MODEL=gemini-2.0-flash
```
Or `gemini-2.0-flash-lite` for higher free-tier quota. The `BRAIN_GEMINI_MODEL` env var overrides the config default without touching `config.py`.

**Note**: The free tier of both `gemini-2.0-flash` and `gemini-2.0-flash-lite` has a daily quota. When exhausted, the harness logs a 429 warning and proceeds with 0 claims (no block on ingestion). Quota resets at midnight Pacific.

## Code Review Audit — 2026-04-16

Full audit of the codebase identified 9 real bugs (2 critical, 1 security, 4 correctness, 2 maintenance). All fixed in one session. Log below for future reference.

---

### Fix 1 — `brain_server.py`: Event Loop Crash on Startup (Critical)

**Symptom**: `brain_server.py` calls `asyncio.run(_initialize())` at module load time to initialize LightRAG. When FastMCP then starts its own event loop, any Ollama client or asyncio resource bound to the first loop is dead. Result: connection errors or `RuntimeError: Event loop is closed` on first query.

**Root cause**: `asyncio.run()` creates and tears down its own loop. Resources initialized inside it cannot be used in a different loop.

**Fix**: Deleted `_initialize()` and `asyncio.run(_initialize())`. Moved RAG initialization into FastMCP's `lifespan` hook, which runs inside the server's own loop:
```python
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(server):
    validate_paths()
    rag = get_rag()
    await rag.initialize_storages()
    logger.info("✅ Brain Bridge is ONLINE.")
    yield

mcp = FastMCP("BrainBridge", lifespan=lifespan)
```

---

### Fix 2 — `watch_archive.py`: Wrong Lock Type for Watchdog Callbacks (Critical)

**Symptom**: Debounce logic uses `asyncio.Lock()` to protect `_pending` and `_last_trigger`, but watchdog fires callbacks on background OS threads. `asyncio.Lock` is not thread-safe — it can only be acquired from within an event loop coroutine.

**Root cause**: `asyncio.Lock` vs `threading.Lock` confusion. Watchdog events are OS threads, not coroutines.

**Fix**: Replaced `asyncio.Lock()` with `threading.Lock()`. Extracted the `fire` boolean outside the lock before calling `_run_index()` (which runs long and must not hold the lock):
```python
self._lock = threading.Lock()

# In _debounce_loop:
fire = False
with handler._lock:
    if handler._pending and elapsed >= handler.debounce_seconds:
        handler._pending = False
        fire = True
if fire:
    handler._run_index()
```

---

### Fix 3 — `query.py`: SQLite Connection Leak

**Symptom**: On any exception inside `search()`, the SQLite connection is never closed. Under load or error conditions this leaks file handles.

**Fix**: Wrapped with `try/finally`:
```python
conn = open_db()
try:
    ...
    return results
finally:
    conn.close()
```

---

### Fix 4 — `index_archive.py`: `--reset` Does Not Actually Reset

**Symptom**: Running `python index_archive.py --reset` clears the manifest JSON but leaves all LightRAG state files intact (`kv_store_*.json`, `*.graphml`, vector stores). LightRAG then loads stale graph data and re-indexes on top of it, creating duplicates.

**Root cause**: WORKING_DIR was never wiped on reset.

**Fix**: Added `shutil.rmtree` pass over WORKING_DIR on `force_reset`, and nulled `_rag_instance` to force a fresh constructor:
```python
if force_reset and WORKING_DIR.exists():
    for item in WORKING_DIR.iterdir():
        item.unlink() if item.is_file() else shutil.rmtree(item)
    _rag_instance = None
```

Also moved `ConnectionError` catch to wrap `initialize_storages()` (where the actual Ollama connection happens), not the pure-Python `LightRAG()` constructor.

---

### Fix 5 — `utils.py`: `shell=True` Subprocess Injection Risk (Security)

**Symptom**: `nvidia-smi` was called via `subprocess.check_output(cmd_string, shell=True)`. If any part of the command string were ever parameterized, this would be a shell injection vector.

**Fix**: Switched to list form with `shell=False`:
```python
cmd = ["nvidia-smi", "--query-gpu=memory.used,memory.total,utilization.gpu", "--format=csv,noheader,nounits"]
output = subprocess.check_output(cmd, shell=False, timeout=5).decode().strip()
```

**Fix**: Corrected to the actual filename:

### Fix 7 — `embed.py`: N+1 DELETEs and Wrong Chunk Count

**Symptom 1**: `_purge_chunks()` issued one `DELETE FROM vec_chunks WHERE rowid = ?` per chunk row. On a file with 50 chunks, that's 50 round-trips to SQLite.

**Fix**: Single batch delete with `IN (...)` placeholder:
```python
placeholders = ",".join("?" * len(row_ids))
conn.execute(f"DELETE FROM vec_chunks WHERE rowid IN ({placeholders})", row_ids)
```

**Symptom 2**: `total_chunks` was incremented by `len(chunks)` (total chunks attempted) instead of `stored_count` (chunks that actually succeeded). The final log line overstated the index size when embedding errors occurred.

**Fix**: Track `stored_count` separately and use that:
```python
total_chunks += stored_count  # was: += len(chunks)
```

### Fix 8 — `news_ingest.py`: O(n) File Scan for Deduplication

**Symptom**: On every run, `_already_ingested()` called `INGEST_PATH.rglob("*.md")` and read every existing file to extract the source URL from its frontmatter. With thousands of articles, this scales O(n) and reads the full content of every file just to deduplicate.

**Fix**: Replaced with a JSON manifest (same pattern as `embed.py` and `fetch_papers.py`). URLs are stored in `data/news_ingest_manifest.json`. Dedup is an O(1) set lookup.

## Key Design Decisions and Why

**Why LightRAG over raw vector DB**: Hybrid search (BM25 + vector + graph traversal) outperforms pure vector search for personal knowledge, especially when queries involve named entities or relationships. Graph traversal finds connections you didn't know to ask for.

**Why nomic-embed-text (768-dim) over larger models**: Fits in RAM, runs fast on CPU, quality is sufficient for semantic retrieval over markdown notes. bge-m3 (1024-dim) would be marginally better but adds latency.

**Why Qwen 3.5:4b over larger models**: Only local model that fits in 6GB VRAM with room for KV cache. Q4_K_M quantization keeps it at 3.4GB. Larger models (7B+) would swap to CPU, destroying throughput.

**Why chunk at 1500 chars**: Math-derived from hardware. See Bug 5 above. Balance between: enough context for meaningful entity extraction, small enough to finish before timeout.

**Why AutoResearchClaw over manual research**: Automated literature search, synthesis, and paper writing in 23 stages. Produces structured knowledge cards alongside the paper. The cards are more useful than the paper for LightRAG indexing because they're atomic and well-scoped.

## For the Next Developer (Human or AI)

If you're reading this to continue the work:

1. **Read `knowledge_base/2. Areas/System_Architecture.md`** first — it has the full Mermaid diagram of how all pieces connect.
2. **The critical invariant**: `index_archive.py` must call Ollama's embed API directly (not via `ollama_embed` from LightRAG). See Bug 4. If you update LightRAG, check if this is still needed.
3. **Chunk size is hardware-coupled**: If you change models or hardware, recalculate using the formula in Bug 5. The 1305-token overhead is from `lightrag/prompt.py`'s `entity_extraction_system_prompt`.
4. **Windows matters**: Everything assumes PowerShell. Env vars are `$env:VAR="value"`. Paths use backslashes in some contexts.
5. **The self-improvement loop is the point**: Don't just use the system — run sprints to improve it. Each sprint should produce a paper that gets indexed, which improves retrieval quality for the next sprint.
